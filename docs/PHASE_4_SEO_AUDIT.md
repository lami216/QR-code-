# Phase 4 SEO audit

**Audit date:** 2026-07-20  
**Starting commit:** `4bcca87873276f62aed0562975e9e2f6a1c31d63`  
**Production origin:** `https://studioqr.online`

## Current architecture before Phase 4

The site used Next.js App Router with a homepage, `/generator`, five specialized tools, three guide detail pages exposed from `/blog`, and route handlers for `robots.txt` and `sitemap.xml`. A shared metadata helper supplied canonical, Open Graph and Twitter fields. Specialized tools used a central typed registry, while guide content used a separate data registry. FAQ, breadcrumb, WebApplication and Article JSON-LD helpers already existed.

## Strengths

- One canonical production origin is normalized in a shared helper.
- Five focused tools have distinct intent, visible explanatory content, metadata and FAQs.
- The generator runs client-side and provides real PNG, vector SVG and PDF exports.
- Sitemap and robots routes are generated by the application rather than maintained as fragile static files.
- Tool and guide data registries provide a sound starting point for controlled expansion.
- Existing schema contains no ratings, testimonials or review claims.

## Weaknesses

- `/generator` was almost entirely the client tool, with a generic `QR Studio` heading and no substantial broad-intent explanation.
- The homepage presented itself as another broad generator landing page, creating avoidable overlap with `/generator`.
- `/blog` acted as a guide list even though detail pages lived under `/guides`; breadcrumb links also pointed back to `/blog`.
- Only three guides existed and the set omitted core questions about WiFi creation, static versus dynamic workflows and logo safety.
- The header did not expose specialized tools, the footer offered only two resource links, and tool-to-guide links were mostly a single generic troubleshooting link.
- Repeated FAQ markup was not encapsulated, and breadcrumb markup was duplicated.
- The sitemap manually repeated guide paths instead of reading the guide registry.

## Missing opportunities

- A stable `/guides` hub with clear topic summaries and direct paths to related tools.
- A deliberate homepage → generator/tools → guides path using descriptive anchors.
- Useful broad-page sections about static behavior, browser privacy, formats, customization, use cases and failure prevention.
- Reusable server components for guide cards, breadcrumbs and FAQs.
- A documented quality gate for future tools so new routes are based on product capability and user need, not keyword variation.

## Competitor-level gaps

Without copying any competitor, mature QR platforms commonly make formats and tool categories easy to discover, explain static/dynamic tradeoffs, publish task-led troubleshooting resources and connect help content to the relevant workflow. QR Studio lacked that cohesive information architecture. It also had no account-based dynamic editing or analytics; those are product choices, not claims the static generator should imitate. The appropriate gap to close now is clarity, testing guidance and navigation—not artificial feature parity or mass landing pages.

## Recommended implementation order

1. Establish `/generator` as the broad tool page while keeping the interface above explanatory content.
2. Separate the homepage role into product introduction, trust and tool/resource discovery.
3. Establish `/guides`, migrate guide navigation to it and publish only the five high-value requested topics.
4. Add contextual tool-to-guide and guide-to-tool links from central registries.
5. Align sitemap, navigation, metadata and accurate visible-content schema.
6. Validate static rendering, internal routes, client generator behavior and performance budgets.
7. Expand only after Search Console and first-party user questions demonstrate a genuine need.
